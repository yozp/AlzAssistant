# 基于大模型的阿尔茨海默症智能问答系统
## 综合课程设计文档

---

## 一、项目概述

本项目是一个基于大语言模型（LLM）的阿尔茨海默症智能问答系统，旨在通过人工智能技术为用户提供专业的阿尔茨海默症相关咨询和初步评估服务。系统采用前后端分离架构，后端使用Spring Boot 3.5.7框架，前端使用Vue 3框架，结合RAG（检索增强生成）技术，能够根据用户输入的症状描述进行智能分析和识别，提供初步评估和专业建议。

### 1.1 项目背景

阿尔茨海默症是一种常见的神经退行性疾病，早期识别和干预对延缓病情发展具有重要意义。传统的医疗咨询方式存在资源有限、响应不及时等问题。本系统通过整合专业的医学知识库和大语言模型，为用户提供24小时在线的智能问答服务，帮助用户了解疾病知识、进行初步症状评估，并提供专业的就医建议。

### 1.2 技术架构

- **后端技术栈**：
  - 框架：Spring Boot 3.5.7
  - 语言：Java 21
  - ORM：MyBatis Flex 1.11.0
  - 数据库：MySQL 8.0+
  - 缓存：Redis（用于Session管理和对话记忆存储）
  - AI框架：LangChain4j 1.1.0
  - 大模型：通义千问（Qwen-Max）、通义千问Embedding（text-embedding-v4）
  - 工具库：Hutool 5.8.38、Lombok、JSoup 1.19.1、Apache POI 5.2.5、Apache PDFBox 3.0.3
  - API文档：Knife4j 4.4.0（基于OpenAPI3）
  - 监控：Spring Boot Actuator + Prometheus
  - 对象存储：腾讯云COS（可选）
- **前端技术栈**：
  - 框架：Vue 3.5.22
  - 语言：TypeScript
  - UI组件库：Ant Design Vue 4.2.6
  - 状态管理：Pinia
  - 路由：Vue Router
  - 构建工具：Vite
  - Markdown渲染：marked + highlight.js
- **核心特性**：RAG检索增强生成、SSE流式响应、多会话管理、知识库管理、AI模型动态切换、对话记忆管理、系统监控

#### 2.1.1.1 用户角色划分

系统包含两种用户角色：

1. **普通用户（user）**
   - 注册和登录系统
   - 创建和管理个人对话会话
   - 与AI进行智能问答
   - 查看对话历史
   - 管理个人信息

2. **管理员（admin）**
   - 拥有普通用户的所有权限
   - 用户管理（增删改查）
   - 知识库管理（上传、编辑、删除文档）

### 2.2 性能需求

1. **响应时间**
   - 用户登录响应时间 < 1秒
   - 页面加载时间 < 2秒
   - AI问答首次响应时间 < 3秒
   - 流式响应延迟 < 500ms

2. **并发性能**
   - 支持至少100个并发用户
   - 系统响应时间在并发情况下不应明显增加

3. **可用性**
   - 系统可用性 > 99%
   - 支持7×24小时运行

4. **数据存储**
   - 支持存储至少10万条对话记录
   - 知识库支持至少1000个文档
   - 单个文档最大支持10MB

5. **扩展性**
   - 支持水平扩展（多实例部署）
   - 支持添加新的大模型
   - 支持扩展新的功能模块

**实体关系说明：**

- **User（用户）** 与 **App（应用/会话）** 是一对多关系（1:N），一个用户可以创建多个会话
- **App（应用/会话）** 与 **ChatHistory（对话历史）** 是一对多关系（1:N），一个会话包含多条对话记录
- **User（用户）** 与 **ChatHistory（对话历史）** 是一对多关系（1:N），一个用户有多条对话记录
- **User（用户）** 与 **KnowledgeBase（知识库）** 是一对多关系（1:N），多个知识库文档由管理员上传
- **User（用户）** 与 **AiModel（大模型）** 是一对多关系（1:N），多个模型配置由管理员创建
- **KnowledgeBase（知识库）** 与 **KnowledgeChunk（知识分块）** 是一对多关系（1:N），一个文档被分割成多个块
- **AssessmentRecord（评估记录）** 与 **SymptomRecord（症状记录）** 是一对多关系（1:N），一个评估包含多个症状
- **App（应用/会话）** 与 **AssessmentRecord（评估记录）** 是一对多关系（1:N），一个会话可以有多条评估记录

### 4.1 登录设计

#### 4.1.1 登录

**功能描述：**
用户通过账号和密码登录系统，系统验证用户身份后创建Session，返回用户信息。

**接口设计：**

**接口地址：** `POST /api/user/login`（注意：实际路径为 `/api/user/login`，因为配置了 `server.servlet.context-path: /api`）

**请求参数：**
```json
{
  "userAccount": "string",
  "userPassword": "string"
}
```

**响应数据：**
```json
{
  "code": 0,
  "data": {
    "id": 1,
    "userName": "用户名",
    "userAvatar": "头像URL",
    "userRole": "user",
    "userProfile": "用户简介"
  },
  "message": "success"
}
```

**实现流程：**
1. 接收用户输入的账号和密码
2. 参数校验（账号和密码不能为空）
3. 查询数据库，验证账号是否存在
4. 验证密码（使用MD5加密后对比）
5. 创建Session，存储用户信息
6. 返回脱敏后的用户信息

**Session管理：**
- 使用Spring Session + Redis实现分布式Session
- Session过期时间：30天
- Cookie过期时间：30天

**相关代码：**
- Controller: `UserController.userLogin()`
- Service: `UserService.userLogin()`
- 实现类: `UserServiceImpl.userLogin()`
- Session配置: `application.yml` 中的 `spring.session.store-type: redis`

#### 4.1.2 注册

**功能描述：**
新用户通过账号、密码和确认密码注册账户，系统验证后创建用户记录并自动登录。

**接口设计：**

**接口地址：** `POST /api/user/register`（注意：实际路径为 `/api/user/register`）

**请求参数：**
```json
{
  "userAccount": "string",
  "userPassword": "string",
  "checkPassword": "string"
}
```

**响应数据：**
```json
{
  "code": 0,
  "data": 1,
  "message": "success"
}
```

**实现流程：**
1. 接收用户输入的账号、密码和确认密码
2. 参数校验（账号长度4-20，密码长度8-20）
3. 验证两次密码是否一致
4. 检查账号是否已存在
5. 密码加密（MD5加盐）
6. 插入用户记录到数据库
7. 返回用户ID

**密码加密：**
- 使用MD5加盐加密
- 密码长度要求：8-20位
- 账号长度要求：4-20位

**相关代码：**
- Controller: `UserController.userRegister()`

### 4.6 RAG检索增强生成

#### 4.6.1 RAG技术实现

**功能描述：**
系统使用RAG（检索增强生成）技术，在生成回答前先从知识库中检索相关信息，确保回答的准确性和专业性。

**实现流程：**
1. 用户输入问题
2. 使用Embedding模型将问题转换为向量
3. 在向量数据库中搜索相似的知识块（相似度阈值0.75，最多返回5个）
4. 将检索到的知识块作为上下文，与用户问题一起发送给大模型
5. 大模型基于检索到的知识生成回答

**技术细节：**
- Embedding模型：通义千问 text-embedding-v4
- 向量存储：InMemoryEmbeddingStore（内存向量存储，适合中小规模知识库）
- 文档分割：按段落分割，最大1000字符，重叠200字符
- 相似度计算：余弦相似度
- 检索策略：Top-K检索（K=5），最小相似度阈值0.75
- 对话记忆：使用Redis存储对话历史（RedisChatMemoryStore），支持多会话独立记忆

**对话记忆实现：**
- 使用LangChain4j的MessageWindowChatMemory实现对话上下文管理
- 每个会话（appId）拥有独立的对话记忆
- 对话历史从数据库加载到Redis，最多保留20条消息
- 支持多轮对话的上下文理解

**相关代码：**
- RAG配置: `RagConfig.java`
- RAG重建服务: `RagRebuildServiceImpl.java`
- 对话记忆配置: `RedisChatMemoryStoreConfig.java`
- AI服务工厂: `AiChatServiceFactory.java`

   - 支持Session管理，基于Redis实现分布式Session

### 7.4 不足与改进

1. **向量存储优化**
   - 当前使用内存向量存储（InMemoryEmbeddingStore），重启后需要重新构建
   - 改进：可以改用Redis向量存储或专业的向量数据库（如Milvus、Pinecone），支持持久化和分布式部署

2. **知识库管理**
   - 当前文档解析和向量生成是同步的，大文档可能耗时较长
   - 改进：可以改为异步处理，使用消息队列（如RabbitMQ、Kafka），提升用户体验

3. **评估功能**
   - 评估记录表和症状记录表已设计，但功能未完全实现
   - 改进：可以完善症状评估功能，提供更专业的评估报告，支持评估记录的查询和统计

4. **性能优化**
   - 当前对话记忆最多保留20条消息，对于长对话可能丢失早期上下文
   - 改进：可以实现更智能的对话记忆管理，如摘要压缩、关键信息提取等

5. **多模态支持**
   - 当前仅支持文本输入和输出
   - 改进：可以支持图片、语音等多模态输入，提供更丰富的交互方式

6. **缓存优化**
   - 当前RAG检索结果没有缓存，相同问题会重复检索
   - 改进：可以实现检索结果缓存，提升响应速度

7. **错误处理**
   - 当前AI模型调用失败时的错误处理可以更完善
   - 改进：可以实现自动重试、降级策略、错误恢复等机制
